# FoodMind 产品需求文档（PRD）

版本：1.1  
状态：MVP 基线  
日期：2026-06-23

## 1. 产品概述

FoodMind 是一个 Android 优先的饮食记录和决策辅助应用。它通过用户自己的饮食历史、口味偏好、预算和可信群组中的饮食记录，帮助用户回答：

> 我今天应该吃什么？

产品重点不是提供餐厅交易服务，而是建立“记录—理解—推荐—再次记录”的个人饮食闭环。

## 2. 问题与机会

目标用户经常遇到以下问题：

- 到了用餐时间不知道吃什么。
- 忘记近期吃过哪些食物，容易重复选择。
- 不清楚自己的饮食支出和偏好趋势。
- 想参考朋友的真实选择，而不是公开平台上的陌生评价。
- 有预算、忌口或饮食目标，但做决定时难以同时考虑。

FoodMind 通过结构化记录和可解释推荐降低选择成本。

## 3. 目标用户

### 3.1 主要用户

- 大学生和校园用户。
- 经常在固定区域用餐的人。
- 希望控制饮食预算的人。
- 希望减少重复饮食或尝试新食物的人。
- 希望参考朋友饮食记录的小型社群用户。

### 3.2 用户能力假设

- 能完成常见 Android 表单操作。
- 能理解价格、评分、辣度、甜度等输入。
- 可能在网络不稳定的环境中使用。
- 不要求用户具备营养或医学知识。

## 4. 产品目标

### 4.1 MVP 目标

1. 用户可以注册、登录并恢复登录状态。
2. 用户可以设置预算、地区、口味和饮食限制。
3. 用户可以快速记录 Meal 和 Drink。
4. 用户可以查看、编辑和删除自己的历史记录。
5. 用户可以创建或加入私密群组。
6. 用户可以查看群组内共享的饮食动态。
7. 用户可以获得带理由的饮食推荐。
8. 用户可以选择、拒绝或将推荐转换为 Meal。
9. 用户可以查看每周和每月饮食分析。

### 4.2 成功标准

- 新用户可以在 3 分钟内完成注册、偏好设置和第一条 Meal 记录。
- 已登录用户可以在 60 秒内新增一条 Meal 或 Drink。
- 用户提交的记录能够立即出现在 History。
- 推荐结果明确展示来源或理由，而不是只显示食物名称。
- 所有核心页面均有 Loading、Empty、Content 和 Error 状态。
- 所有 API 错误都有用户可理解的反馈，不出现静默失败。

## 5. 非目标

MVP 不包含：

- 外卖下单、支付和配送。
- 公共社交网络、关注和私信。
- 完整餐厅目录和地图搜索。
- 医疗、减肥或专业营养诊断。
- 实时聊天。
- 图片上传和图片识别。
- 离线数据库同步。
- 推送通知。
- AI 对话式助手。

## 6. 核心产品闭环

```text
注册或登录
  → 完成个人偏好
  → 记录 Meal / Drink
  → 查看个人历史
  → 创建或加入 Group
  → 查看可信群组动态
  → 请求 Recommendation
  → 选择、拒绝或标记已食用
  → 查看 Analytics
  → 调整下一次饮食选择
```

## 7. 功能需求

### FR-01 注册

用户故事：

> 作为新用户，我希望通过邮箱创建账号，以便保存自己的饮食数据。

规则：

- 邮箱必填且格式有效。
- 密码至少 8 个字符。
- 用户名长度为 2～80 个字符。
- 邮箱已存在时显示明确错误。
- 注册成功后保存 JWT。
- 注册成功后进入 Profile Setup，不直接进入 Home。

验收条件：

- 合法输入返回成功并进入 Profile Setup。
- 非法字段在对应输入框显示错误。
- 请求失败时保留已输入内容。
- 重复点击提交不会产生多个账号或多个并发请求。

### FR-02 登录和会话恢复

用户故事：

> 作为已有用户，我希望登录后保持会话，以便下次打开应用时直接继续使用。

规则：

- 登录成功后保存 JWT。
- App 启动时，如果存在 JWT，则调用 `/api/auth/me`。
- Token 无效或过期时清除本地会话并进入 Login。
- 用户主动退出时清除 Token 和本地用户状态。
- Profile 未完成时进入 Profile Setup，否则进入 Home。

验收条件：

- 正确账号可登录。
- 错误账号信息显示“邮箱或密码错误”。
- 有效 Token 可恢复会话。
- 任何受保护接口返回 401 后，应用只执行一次统一登出。

### FR-03 个人偏好

用户故事：

> 作为用户，我希望设置预算、口味和忌口，以便推荐更符合我的需要。

字段：

- 最低预算和最高预算。
- 辣度容忍度。
- 常用地区。
- 饮食目标。
- 默认隐私级别。
- 喜欢和不喜欢的菜系。
- 饮食限制。

规则：

- 最低预算不得大于最高预算。
- 辣度范围为 0～5。
- 同一菜系不能同时出现在喜欢和不喜欢列表。
- 首次设置成功后标记 Profile 已完成。
- 编辑已有 Profile 时先加载服务器值。

### FR-04 新增 Meal

用户故事：

> 作为用户，我希望记录吃过的食物，以便建立个人饮食历史。

必填字段：

- `foodName`
- `mealType`
- `privacyLevel`
- `consumedAt`

规则：

- `price >= 0`。
- `rating` 为 0～5。
- 选择 `GROUP` 时必须选择一个已加入的群组。
- 用户没有群组时，禁用 `GROUP` 选项并说明原因。
- 创建成功后进入 History，并显示新增记录。
- 创建失败时保留表单。

### FR-05 新增 Drink

与 Meal 相同，但使用 Drink 字段：

- 饮品名称。
- 店铺名称。
- 甜度。
- 冰量。
- 价格、评分、评论。
- 是否愿意再次购买。

### FR-06 历史记录

用户故事：

> 作为用户，我希望查看和管理过去的饮食记录。

规则：

- Meal 和 Drink 可以通过标签页切换。
- 默认按 `consumedAt` 从新到旧显示。
- 支持关键词和基础筛选。
- 支持分页加载。
- 点击项目进入详情。
- 用户只能编辑和删除自己的记录。
- 删除前必须确认。
- 删除成功后立即从列表移除。

### FR-07 群组

用户故事：

> 作为用户，我希望与认识的人共享饮食记录，以便参考可信朋友的选择。

规则：

- 用户可以创建群组并获得邀请码。
- 用户可以通过邀请码加入群组。
- 群组列表显示名称、角色和成员数量。
- 只有 ACTIVE 成员可以查看详情、成员和 Feed。
- Feed 只显示 `privacyLevel=GROUP` 且属于该群组的记录。
- Feed 可以混合显示 Meal 和 Drink。
- OWNER/ADMIN 可以移除成员，普通 MEMBER 不可以。

需后端确认：

- MEMBER 主动退出群组的接口。
- OWNER 解散群组或转移所有权的规则。
- ADMIN 是否可以移除其他 ADMIN。

### FR-08 推荐

用户故事：

> 作为用户，我希望基于历史、偏好和朋友选择获得三个可解释的推荐。

规则：

- 用户输入 Meal Type、预算、地区等条件。
- 成功时优先展示三条推荐。
- 每条推荐显示名称、餐厅、类型、得分和理由。
- 用户可以选择、拒绝或标记已食用。
- 拒绝时可以填写原因。
- “已食用”会创建 Meal 记录。
- 数据不足时显示专门的 Empty State，而不是通用错误。

推荐标签：

| API 值 | UI 标签 |
|---|---|
| `SAFE_OPTION` | Safe |
| `EXPLORE_OPTION` | Explore |
| `FRIEND_PICK` | Friend Pick |

### FR-09 Analytics

用户故事：

> 作为用户，我希望了解近期消费和重复饮食情况，以便改善下一次选择。

规则：

- 默认展示 Weekly Analytics。
- 可查看 Monthly、Repetition 和 Spending。
- 无数据字段必须安全显示为 `—` 或空状态。
- 金额统一格式化为两位小数。
- 图表不是 MVP 强制项；摘要卡片和列表足以满足功能。

### FR-10 Home

Home 是信息摘要和功能入口，不承担全部业务操作。

建议内容：

- 用户问候。
- 本周 Meal/Drink 数量。
- 本周总支出。
- 今日推荐入口。
- 快速记录 Meal/Drink。
- 最近记录。

Home 数据加载失败时，各模块应独立降级，避免整个页面完全不可用。

## 8. 页面入口与信息架构

登录后的主功能区包含五个一级入口：

```text
Home | Log | History | Groups | Profile
```

二级入口：

- Recommendation：从 Home 进入。
- Analytics：从 Home 或 Profile 进入。
- Meal/Drink Detail：从 History 或 Group Feed 进入。
- Group Detail/Feed：从 Groups 进入。

登录、注册和首次 Profile Setup 不显示底部导航。

Android 课程实现采用多 Activity 架构。五个一级入口分别由独立 Activity 承载；底部导航仅作为统一入口控件，点击后通过显式 Intent 打开目标 Activity。产品需求不依赖 Fragment、NavHost 或 Navigation Component。

## 9. 全局状态要求

每个远程数据页面必须支持：

| 状态 | 行为 |
|---|---|
| Initial | 尚未请求，可显示占位或空白 |
| Loading | 显示加载状态并防止重复提交 |
| Content | 显示有效数据 |
| Empty | 显示场景化空状态和下一步操作 |
| Error | 显示错误说明和 Retry |
| Refreshing | 保留已有内容并显示刷新状态 |
| Submitting | 保留表单并禁用提交按钮 |

## 10. 非功能需求

### 10.1 可用性

- 可点击区域不小于 48dp。
- 表单字段使用清楚的 Label，不仅依赖 Hint。
- 错误信息靠近对应字段。
- 危险操作需要确认。
- 返回操作不得意外丢失已编辑表单；有修改时应提示。

### 10.2 性能

- 普通页面首次加载目标小于 2 秒（正常网络条件）。
- 列表使用 RecyclerView，不一次加载全部数据。
- 防止页面重复触发相同 API。

### 10.3 安全

- 不在 Logcat 输出 JWT、密码或完整认证响应。
- Token 仅保存在应用私有存储。
- 所有受保护接口通过 Bearer Token 调用。
- 生产环境必须使用 HTTPS；HTTP 仅限本地开发。

### 10.4 兼容性

- Kotlin + AndroidX。
- XML Layout。
- 最低 Android 版本由课程要求决定，建议 `minSdk 24`。
- 支持常见手机竖屏尺寸。
- MVP 不强制支持平板和横屏专门布局。

## 11. MVP 完成定义

MVP 完成必须满足：

- 核心闭环可以在真实后端上完整运行。
- 所有 P0 功能通过验收测试。
- 无导致崩溃的已知核心路径问题。
- 401、无网络、500、空列表均有处理。
- 不使用硬编码用户 ID、群组 ID 或假 Token。
- 所有 enum 与 API 字符串严格一致。
- 项目能够从干净环境构建和运行。
